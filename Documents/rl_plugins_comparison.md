# UE5 強化学習プラグイン・訓練方法の比較

## 概要

UE5で強化学習を行う場合、主に3つのアプローチがあります。  
本ドキュメントでは各アプローチの特徴・メリット・デメリット・並列訓練対応状況を整理します。

---

## アーキテクチャ比較

### 1. gRPC（自作通信）

```
Python (RLlib / SB3)
├── Worker 1 ↔ UE5 Instance 1
├── Worker 2 ↔ UE5 Instance 2
└── Worker 3 ↔ UE5 Instance 3
         ↓（ONNX export）
       UE5 C++ 推論（NNI）
```

**メリット**
- RLフレームワーク（SB3, RLlib, CleanRL等）を自由に選択可能
- 観測データの形式・通信頻度を完全に制御できる
- 高速な双方向通信

**デメリット**
- UE5側にgRPCプラグインの導入が必要（公式サポートなし）
- 通信プロトコルの設計・実装コストが高い
- 複数インスタンスによる並列化はリソースを大量消費

**並列訓練**  
複数のUE5インスタンスを起動し、Pythonの `SubprocVecEnv` 等で並列化。  
インスタンス数分のGPU・メモリが必要。

---

### 2. UnrealCV

```
Python (Gym互換環境)
├── UE5 Instance (Port: 9000)
├── UE5 Instance (Port: 9001)
└── UE5 Instance (Port: 9002)
         ↓（ONNX export）
       UE5 C++ 推論（NNI）
```

**メリット**
- Python APIが整備済みで導入が比較的簡単
- 画像ベースの観測取得に強い（カメラ・深度・セグメンテーション）
- OpenAI Gym との連携実績あり

**デメリット**
- UE5.4 対応が不確実（コミュニティ維持）
- 画像ベース前提の設計で数値ベース観測には過剰
- 更新頻度が低下傾向にある

**並列訓練**  
複数UE5インスタンスをポート分けて起動することで並列化可能。  
ただし公式の並列管理機能はなく、自前実装が必要。

---

### 3. Learning Agents（採用）

```
UE5 エディタ（1プロセス）
└── ULearningAgentsManager
    ├── Agent 1
    ├── Agent 2
    ├── Agent 3
    └── ...（N個）
         ↓
   ULearningAgentsPolicy（NNI）
```

**メリット**
- Epic 公式プラグインで UE5.4 対応が確実
- **単一UE5インスタンス内で複数エージェントを並列訓練可能**
- 観測・行動・報酬の定義がC++ APIとして整備済み
- エディタ上で訓練の可視化・デバッグが可能
- 推論はNNIプラグインでシームレスに移行できる

**デメリット**
- 学習アルゴリズムのカスタマイズに制約がある
- 公式ドキュメント・事例がまだ少ない

**並列訓練**  
`ULearningAgentsManager` に複数エージェントを登録するだけで並列化。  
単一プロセスで完結するためリソース効率が圧倒的に高い。

## 一覧比較表

| 観点 | gRPC | UnrealCV | Learning Agents |
|------|:----:|:--------:|:---------------:|
| 導入コスト | 高 | 中 | 低 |
| UE5.4 対応 | 自前で対応 | 不確実 | 確実（公式） |
| RLライブラリの自由度 | 高 | 高 | 中 |
| 数値ベース観測 | 適 | 不向き | 適 |
| 画像ベース観測 | 実装次第 | 最適 | 可能 |
| 並列訓練方式 | 複数インスタンス | 複数インスタンス | 単一インスタンス内 |
| 並列訓練の実装コスト | 高 | 高 | 低（標準機能） |
| リソース効率 | 低 | 低 | 高 |
| 実機推論への移行 | 手動でONNX連携 | 手動でONNX連携 | NNIで統合済み |
| 保守性 | 自己責任 | コミュニティ依存 | Epic公式 |

## 訓練から推論への流れ（共通）

```
[訓練]                          [実機推論]
Python / UE5 で訓練
        ↓
  モデルをONNX形式で出力
        ↓
UE5 NNI（Neural Network Inference）プラグインで読み込み
        ↓
  C++ から推論を呼び出す
```

訓練時のみPython通信や外部プロセスが必要で、  
**実機では UE5 C++ のみで推論が完結します。**

---

## 本プロジェクトの選択

**Learning Agents** を採用。

理由：
1. バランスゲームは数値ベースの観測（角度・速度・位置）が主であり、UnrealCVの画像ベース設計は不要
2. 単一インスタンス内での並列訓練により、複数エージェントを効率よく同時訓練できる
3. Epic公式サポートで UE5.4 対応が確実
4. 推論への移行がNNIプラグインでシームレスに行える

### 実装クラス構成

| クラス | 役割 |
|--------|------|
| `ABalanceCart` | カート + 2本のポール（物理制約で接続） |
| `UBalanceInteractor` | 観測6値と行動1値の定義 |
| `UBalanceTrainer` | 報酬関数・エピソード終了条件の定義 |
| `ABalanceTrainingManager` | 訓練ループ管理・エージェント登録 |

### 観測空間

| 観測値 | 説明 |
|--------|------|
| CartPosition | カートのX位置（正規化済み） |
| CartVelocity | カートのX速度（正規化済み） |
| LowerPoleAngle | 下部ポールの傾き角度（正規化済み） |
| LowerPoleAngularVelocity | 下部ポールの角速度（正規化済み） |
| UpperPoleAngle | 上部ポールの傾き角度（正規化済み） |
| UpperPoleAngularVelocity | 上部ポールの角速度（正規化済み） |

### 行動空間

| 行動値 | 説明 |
|--------|------|
| CartForce | カートに加える力（-1〜1、左右） |

### 報酬設計

- ポールが立っている毎ティック: **+1.0**
- 倒れたらエピソード終了（`FallAngleThreshold` = 60度）
